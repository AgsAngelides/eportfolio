Secure Software Development


Introduction to the module

The Secure Software Development modules aims to provide a detailed overview of programming languages and risks associated to those, from the perspectives of design and architecture approaches, programming paradigms. It will cover the subject of software testing and it will look into the impact the Operating Systems and Libraries have on software development. In addition to the programming languages, topics such as the distributed systems, APIs, future trends in secure software development, etc. will also be addressed.

In particular, this module will:

  • Introduce the concept of abstraction of programming.
  • Develop an understand of the basic principles of secure software development methodologies.
  • Explore how to perform analysis, program design, software construction and testing required for secure software development.
  • Introduce the basic concepts of software architecture and Software Development Life Cycles such as Agile and TOGAF.
  • Further develop team working skills.
  • Provide an opportunity to reflect on personal development.

Unit overview

The Unit 1 focusses on the review of various software developing methodologies. In this unit we have reviewed the Waterfall and Agile as well as Secure Agile software development approaches as well as looked into the Unified Modelling Language (UML) and how this could support software development. During this week we have also touched upon the subject of various software development standards applicable.

Please note, the team contract signatures were excluded for privacy conciderations.

For this discussion post I have chosen to analyse a vulnerability from the OWASP 2017 Top 10 Injection category (Mitre, 2017) – OS Command Injection.

The OS Command Injection is a vulnerability concerned with an unauthorised person being allowed to execute commands directly on the operating system. This is usually done via a third-party system, such as a web application, which uses certain commands to access the OS (Mitre, 2022).

To visualise the actions taken by the development team and the attacker that could have led to an attack abusing this weakness, the Unified Modelling Language (UML) Sequence diagram has been chosen. This type of diagram captures the process flow and interactions between different interacting parties (Seidl, 2015) and thus it was felt that it was the most suitable UML diagram in this instance.

The sequence presented in the Figure 1 reviews an OS Command Injection vulnerability example introduced by the usage of the Python OS module and, specifically, its os.system() method. The web application uses this method to fetch the user’s email address entered in the front end, pass it to the OS and then to trigger an email to the user.

Using the os.system() method in Python on a web application may provide an attacker with an opportunity to inject malicious payload which could execute code remotely (Turcsányi, 2022) on the webserver or download internal configuration files, as demonstrated in the example.

During week 1 of the 'Secure software development' module we have been asked to complete a variety of reading, prepare a UML diagram on the assigned subject as well as to write and sign a team contract for the newly allocated team.

This week's reading material included an overview of the OWASP Top Ten (Mitre, 2017), the introduction into a new software development methodology called Secure Scrum (Pohl & Hof, 2015), ISO 27000 standards (ISO/IEC, 2018), software design patterns (Asghar et al, 2019) and an introduction into software development using Python as well as security considerations the developers must have when using this programming language (Pillai, 2017).

I enjoyed how the authors of the Secure Scrum paper managed to utilise a currently popular software development approach for security related issue development work. The paper not only discussed the theoretical benefits of such methodology, but also practical means of implementing it and provided some great examples to further explain their ideas.
I found this paper most relevant to my current role as the Project Manager/Business Analyst. Depending on the type of project, we may deliver some using the waterfall methodology and others using the Agile methodology. When using the Waterfall, I always make sure that in the project design phase we cover the security aspects. However, when using the Agile method I find that the security aspects are often missed during the many iterations or you start off with having them, but in the subsequent iterations you loose the sight of it. The suggestions made in the paper on using the Security tags to tag to the issues in the iteration which then link to the security vulnerabilities is a great way of not loosing the track of the security exposures and tracking them to completion along many sprints.

The other reading material was interesting too, however, some of it was a little less relevant to my current job, although, still informative.

The first module seminar we had this week was also very good and informative. The assignment requirements discussed by the tutor have cleared some things up for me. We have also been advised we would be assigned to the teams of 3 to 4 people in which we will conduct all future assignments. I have also requested the tutor to let us know if any seminar preparation must be done in teams going forward since it was not always clear in the previous module.

During the Unit 1 we have also been asked to prepare an initial discussion forum post on a chosen OWASP Top 10 vulnerability (Mitre, 2017). The post had to include a Unified Modelling Language (UML) diagram of our choice to best represent a chosen coding weakness and the steps that led to it occuring.
I have chosen to use the UML Sequence diagram, because I felt that this type of the diagram was most suitable to represent the actions flowing from one system/actor to the other. The task was interesting since it made me think more of a particular vulnerability I chose from the list. It made me understand what coding practices and actions taken by the user could lead towards exploiting this vulnerability.

Lastly, in this unit we were assigned to our team and asked to create a team contract. It didn't take us long to get going, when compared to the previous module with team work, as I think most of us already knew how to approach the team work in the online learning environment. During this week we have also discussed the domain we wish to work on in our Unit 3 and 6 assignments as well as considered any other upcoming team tasks. A Google drive folder was created for storing all the team related work and team meeting availability was also discussed. The team was very responsive and keen to get started.

Unit overview

The aim of the Unit 2 is to showcase how Agile development stages could be integrated with secure programming practices as well as to discuss the instances in which the software security can be problematic.

For this seminar, we have been asked to read an article by Sharma & Bawa (2020) on secure Agile development and to build table with Scrum development phases and applicable security activities that could be undertaken for each. Please find the table below.

Scrum Agile software development stages secure software development processes to apply
Concept Prepare a role matrix to identify all user types and authorisation level for each (With the help of CLASP and Microsoft’s SDL).
Define the security policy to be followed (i.e. if it should adhere to the client’s or internal policy) and any governing regulations to consider (With the help of CLASP, SDL, CT, and CC.)
List other systems the new software will interact with and whether they are trusted/not trusted (With the help of CLASP).
Define the operating environment assumptions (With the help of CLASP).
Define any 3rd party systems the software/application will interact with and whether they are trustworthy or not (With the help of CLASP).
Inception Assign a security specialist to the project, if one is available.
Perform risk/vulnerability analysis. Provide details of how each will be resolved (With the help of CT and CC).
Define the specific security functionality required to meet the security policies and regulations requirements.
Decide on any secure coding rules, practices that the teams should follow during the iterations (With the help of SDL).
Define each security requirement acceptance criteria.
Define reference security architecture (AA)
Iteration Re-evaluate the list of vulnerabilities as there may be new introduced by the development approach or new identified due to the changes in the requirements.
Tag previously identified security vulnerabilities to relevant sprints for resolution.
Evaluate the security mitigations implemented against the previously agreed acceptance criteria.
Employ pair programming practices were possible to ensure the code is written by following the secure software development practices.
Release Test for all previously identified security vulnerabilities/risks.
If any new security bugs/vulnerabilities are discovered, address them before the deployment.
Retest to confirm remediation.
Perform pen testing on the application and network where it will be implemented (if possible) (With the help of CT).
Document the security architecture.
Prepare a security incident plan: a checklist to follow when a security incident occurs.
Prepare the security improvement plan.
Maintenance Patch all the systems promptly.
Implement any security improvements where feasible.
Offer training to users, including security aspects to consider.
Review project, resolve any newly identified bugs.
New iterations taking place can turn existing projects/products into upgraded versions of it including additional features.
Pentesting at least annually.
Retirement Ensure the data is disposed of or stored in a way that meets the regulatory requirement, e.g. consider storage security, authorisation to access the data (Tolson, 2016), etc.
Cut off any hardware or networking resources.
Migrating users to another application will occur in case of a replacement and security considerations relevant should be addressed in case of this approach.

Various Software Development processes used in the table above are described below:

  • Microsoft’s Secure Development Lifecycle (SDL): increases reliability and reduces maintenance cost.
  • Comprehensive Lightweight Application Security Process (CLASP): Enhances Security.
  • Cigital Touchpoints (CT): improves security and quality of product.
  • Common Criteria (CC): allows the user of the computer system to specify security assurance and functional requirements.
