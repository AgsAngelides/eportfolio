For this post I have chosen to create an ADTree analysing the home router vulnerabilities (see Figure 1). Because of the growing interest in smart homes and constantly expanding IoT, the home router security has become very important (Ray & Bagwari, 2020).

The list of vulnerabilities in Figure 1 has been prepared by conducting qualitative research of various recently reported home router security vulnerabilities, however, it should be noted that the this list does not encompass all potential security risks (Kost, 2022; Horowitz, 2014; Avast, 2022; Wagenseil, 2022).

To analyse these risks quantitively, the Attributes Domain functionality provided by the ADTree tool was used, namely, the ‘Probability of success’ option, which was chosen due to being able of providing an estimated view over the likelihood of the event happening.

Each leaf with yellow background relevant to the attacker was manually assigned an estimated value as follows:

  • 1 – the attack does not require experience, very easy to carry it out.
  • 0.5 – the attack can be carried out by inexperienced attacker.
  • 0 – the attack can be carried out by experienced attackers; unlikely to impact home network.

Similarly, each countermeasure (square green boxes in Figure 1) has also been manually assigned a value based on how easy/difficult it is to implement the measure:

  • 1 – very easy to implement; no computing/networks experience is needed.
  • 0.5 – some computing experience or access is needed.
  • 0 - very difficult for an average home user to implement.

In the latter evaluation, the user computer skill research in rich countries (Nielsen, 2016) was used to determine the value to assign: the study suggests that only 5% of adult papulation can define a problem and use tools for this problem’s resolution.

The ADTree tool then calculated the resulting overall probability of success for abusing the home router based on the vulnerabilities and countermeasures identified.

In relation to echo-server.py, what is achieved using the command: s.bind((HOST, PORT))?

We have specified two variables in the python script, one for the host (localhost) and one for the port (a non priviledged port). Unit overview

The aim of this module is to create a producer-consumer implementation scenario using Python and to research the literature to identify the Quality of Service requirements of distributed applications.

A protocol is a system which uses particular rules, syntax, semantics and synchronization to allow multiple devices to transmit information between them (comptia, 2022).

In a smart home network protocols are responsible for passing the information between the controller hub, the IoT devices and often the mobile application which provides user interface to managing it all, often whilst away from the home.

Let us consider a smart home system comprising of the controller hub, smart door lock, smart thermostat and smart doorbell. There are a variety of protocols that are currently used to support communications between the IoT devices which often have limited memory and battery, use low end microcontrollers and so there is a need for better more lightweight network protocols (Lin & Bergmann, 2016).

6LoWPAN is a lightweight open standard network protocol defined by the Internet Engineering task force and it provides a way of carrying data packets in the form of IPv6 over IEEE 802.15.4 wireless networks, which in turn permits direct connectivity to a wide variety of networks and the internet (Seliem & Elgazzar, 2018).

This protocol uses a strong AES-128 encryption mechanism (Glissa & Meddeb, 2019), it compresses the protocol stack headers, enables stateless addressing, provides consistency with the upper layers (Hennebert & Santos, 2014).

Since the protocol operates over the IEEE 802.15.4, it benefits from this wireless networks data encryption and authentication (Hennebert & Santos, 2014).

That said, some additional security measures are required to ensure the end-to-end security. This could be done by implementing a compressed IPsec network layer protocol developed specifically for 6LoWPAN to ensure data is secure when being transmitted over the network (Hennebert & Santos, 2014).

In addition, an application layer protocol MQTT (which is compatible with 6LoWPAN) could be used to provide a reliable, secured multicast TLS/SSL security. It allows many messages to be transmitted and does not consume a lot of power which supports the Confidentiality, Integrity and Availability principles (Biju & Shekokar, 2017).

With the podcasts becoming more popular, Software as a Service (SaaS) podcast applications have seen a large increase in their user base (Bate, 2020) over the last few years. Since many smaller companies tend to use a monolithic architecture (Kwiecen, 2022), it is safe to assume that many of the podcast SaaS applications started off as monoliths too. This leads us to speculate that due to the growing user base, it could become increasingly harder to manage such applications and a different approach is required.

Monolithic SaaS applications, such as those used for podcasts, are usually composed of a webserver, a load balancer, a catalogue of podcasts, a payment system (for subscription) and a database (talend, 2022).

Breaking down a monolith requires experienced architects and developers and can be costly; microservices can also be bad for the business if implemented incorrectly, for example, if microservice boundaries are worked out wrong (Newman, 2019). It can also be complex to operate and troubleshoot (Newman, 2019).

However, at the same time it can improve the team’s autonomy, reduce the time taken to deploy the new features, provide cost effective scaling, lessen the impact of outages, etc (Newman, 2019).

The breakdown of the monolith could begin with building the underlaying architecture, continuous delivery pipelines and APIs (Fowler, 2018), for example, it could include building the user authentication service first: the edge service which is loosely coupled. This could be followed by other items, such as the podcast subscriptions, podcast notifications, payments, etc.

Next, the data in the monolith database should be decomposed to work with the new microservices (Behara, 2018), i.e. it should follow a domain driven design. For example, we may want to store the podcast payment information in the database separate from the one which stores the podcast author information.

The end product could contain of multiple microservices for managing the payments, users, podcast authors, podcast posting, podcast subscription, podcast downloads for offline listening, etc.
